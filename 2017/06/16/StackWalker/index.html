在Release模式下输出调用堆栈

发表于 2017-06-16 | 分类于 C/C++

1. 前言

调试的时候经常会用到调用堆栈，用VC的话在Debug模式下，碰到异常问题都会停下来，就可以直接看到调用堆栈，但在Release模式下就做不到了。在网上百度了很久，发现可以用MS提供的API(StackWalker)做到这一点，但网上的资料也不是很全，使用的过程中也碰到不少问题，这里记录一下。

2. 实现

3. 问题

其实代码实现还算比较简单的，但实际使用中碰到了各种头疼的问题，花了很长时间才一一解决了，这里记录一下。

a. 函数使用

  • 看了一些网上帖子，经常有把StackWalk，SymGetSymFromAddr，SymGetLineFromAddr三个函数与其64位的形式(SymGetLineFromAddr64)混合使用，建议全部直接用原文中的宏(函数原名)，可以同时兼容32位和64位的程序。
  • IMAGEHLP_SYMBOL，IMAGEHLP_LINE中的SizeOfStruct一定要分别等于sizeof(IMAGEHLP_SYMBOL)和sizeof(IMAGEHLP_LINE)，否则SymGetSymFromAddr和SymGetLineFromAddr调用失败返回错误代码。
  • SymGetLineFromAddr碰到错误代码487 ，表示访问无效地址，一般是缺失pdb文件造成的。

    b. 编译链接

    debug版就不详说了，这里主要针对Release版。要拿到代码文件和代码行信息，就必须生成pdb文件。
  • 把Linker->Debugging->Generate Debug Info设为 Yes(/DEBUG)。
  • 把C/C++->General->Debug Information Format设为Program Database(/Zi)。
  • 如果得到的调用堆栈信息不全，中间很多函数都省略掉了，可以考虑关掉优化(/Od)，把C/C++->Optimization->Optimization设为Disable(/Od)。

a. SetUnhandledExceptionFilter

对于很多crash错误，一般由未被正常捕获的异常引起。在程序crash之前，windows操作系统会默认调用UnhandledExceptionFilter来简单处理这个异常。除此，windows提供了一个API函数可以让用户自定义这里这个处理异常的函数，就是SetUnhandledExceptionFilter。用法：
先仿造UnhandledExceptionFilter来自己定义一个异常处理函数：

pExceptionInfo包含了异常发生时异常线程的状态，返回值有3种：
EXCEPTION_EXECUTE_HANDLER：表明异常处理完毕，程序可以退出
EXCEPTION_CONTINUE_EXECUTION：忽略此异常，或者说处理完此异常，从异常点继续运行。如果此时再发生异常，还会调用异常处理函数。
EXCEPTION_CONTINUE_SEARCH：第三种就是调用默认的方法，可能就是弹出一个框，当然这个默认的方法通常是遵循SetErrorMode的模式。

值得注意的是，UnhandledExceptionFilter，以及SetUnhandledExceptionFilter设置的异常处理函数为全局的处理函数，是异常发生以后，最后一次处理异常的机会。如果程序被调试器附加，调试器将优先于上述异常函数处理异常；程序中try{}catch{}和 _try()_catch{}也将会优先处理异常。
